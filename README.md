# SpeedWave

Автоматизация развёртывания обратного прокси на базе **NGINX + XRAY**, а также установки панели управления и ноды **Remnawave**. Xray слушает порт `443` напрямую и передаёт трафик через Unix-сокет в NGINX — это снижает накладные расходы TCP и повышает надёжность соединений.

> [!CAUTION]
> Репозиторий — **учебный пример** для изучения NGINX, реверс-прокси и основ сетевой безопасности. **Не для продакшена.** Используйте на свой страх и риск. Применение для обхода блокировок/цензуры в ряде стран незаконно; автор не несёт ответственности за последствия.

## Особенности

**SpeedWave** — установщик панели и ноды **Remnawave** с обратным прокси NGINX + XRAY. Ключевые особенности:

- **Поддержка Ubuntu 26.04 LTS (Resolute Raccoon).** В оригинале установка падала на 26.04 — добавлено кодовое имя `resolute` в проверку ОС.
- **Встроенный «Ускоритель ноды» (node-accelerator).** Новый пункт меню: оптимизация ядра/сети (XanMod + BBRv3, sysctl, RPS/RFS, лимиты, swap), фаервол (nftables: анти-скан/флуд + CrowdSec) и диагностика. Локализован (RU/EN), с возможностью отката. Движок — [node-accelerator](https://github.com/jestivald/node-accelerator) (MIT), адаптированный под этот скрипт.
- **Новый дизайн терминала в стиле Claude.** Тёплая коралловая палитра, шапка в рамке, единообразные меню. Цвета автоматически отключаются при выводе в файл/пайп.
- **Только Nginx.** Веб-сервер Caddy убран — установка всегда на Nginx (без лишнего выбора).
- **Один шаблон вместо случайных.** Вместо набора случайных selfsteal-шаблонов — одна страница «IP заблокирован», мимикрирующая под автоматическую блокировку хостинг-провайдера (self-contained, без внешних загрузок).
- **Установка ноды без selfsteal-домена.** Добавлен режим «только нода» — без маскировочного сайта, домена и TLS-сертификата (полезно, когда Reality использует внешний dest).
- **Сборка ядра ноды (rw-core / Xray) из исходников.** Отдельный пункт главного меню: собирает свежий [XTLS/Xray-core](https://github.com/XTLS/Xray-core) **из последних коммитов репозитория** (не из релиза) и подменяет им штатный core в контейнере remnanode через `docker-compose.override.yml`. Проверка обновлений по коммиту, откат к штатному ядру, авто-очистка всех артефактов сборки (Go-тулчейн, исходники, кэши) после установки.

> [!IMPORTANT]
> Поддерживаются Debian (bullseye / bookworm / trixie) и Ubuntu (jammy / noble / resolute). Скрипт тестировался на виртуализации KVM. Для режима с selfsteal-сайтом нужен собственный домен; режим «только нода» домена не требует. Запускать от root на свежей системе. Под Docker и образы нужно ~2.5 ГБ свободного места — на маленьких VPS увеличьте диск заранее (при нехватке установка Docker выдаст понятную ошибку про диск).

## Ускоритель ноды (node-accelerator)

Пункт **7** главного меню. Подменю:

- **Оптимизатор** — ядро XanMod (BBRv3), sysctl-тюнинг (буферы под объём RAM, conntrack, anti-spoof, syncookies), RPS/RFS/XPS, лимиты `nofile/nproc`, swap (zram на мелких нодах), THP off, CPU governor, NIC tuning. Идемпотентно.
- **Защита** — фаервол на nftables (своя таблица `na_filter`, не флашит чужие правила): анти-скан, anti-spoof, защита от SYN/UDP-флуда и ssh-флуда, интеграция с CrowdSec.
- **Диагностика** — read-only отчёт о состоянии ноды.
- **Всё сразу** — оптимизация → защита → диагностика.
- **Откат** — снять оптимизатор и/или фаервол (бэкапы остаются в `/var/backups/node-accelerator/`).

После применения на ноде остаётся read-only команда `na-diagnose` (стабильный вывод для мониторинга/панелей).

> [!NOTE]
> Кастомное ядро XanMod ставится только на x86_64 и не в контейнере (openvz/lxc делят ядро хоста). После установки ядра нужна перезагрузка, чтобы BBRv3 активировался. Кэш apt после установки ядра автоматически чистится, чтобы не забивать диск.

## Обновление ядра ноды (rw-core / Xray)

Пункт **8** главного меню. Внутри контейнера remnanode прокси-движок — это [XTLS/Xray-core](https://github.com/XTLS/Xray-core), лежащий по пути `/usr/local/bin/xray` (симлинк `rw-core`). Этот модуль собирает Xray-core **из последних коммитов репозитория** (а не из готового релиза) и подменяет им штатный core, не трогая основной `docker-compose.yml`.

- **Собрать из последних исходников и установить** — скачивает Go-тулчейн во временную папку, клонирует ветку `main`, собирает бинарник, монтирует его в контейнер через `docker-compose.override.yml` и пересоздаёт ноду. Записывает установленный коммит в `/opt/remnanode/.rw-core-version`.
- **Проверить исходники** — сравнивает установленный коммит с последним в репозитории и пересобирает, только если появились новые правки.
- **Откат к штатному ядру** — удаляет override и собранный бинарник, возвращая core из образа.

Все артефакты сборки (Go-тулчейн ~250 МБ, исходники, кэши Go) удаляются сразу после установки. Перед сборкой проверяется свободное место (нужно ~1.2 ГБ во временной `/var/tmp`); при нехватке сборка не запускается с понятным сообщением.

> [!NOTE]
> Сборка из исходников требует места под Go-тулчейн и кэши. На VPS с маленьким диском (≤ 4 ГБ) освободите место заранее. Поддерживаются архитектуры x86_64 (amd64) и arm64.

## Шейпер трафика (eBPF + EDT)

Пункт **9** главного меню. Ограничитель скорости **на каждого пользователя (по IP)**, работающий в ядре через **eBPF + EDT** (Earliest Departure Time) — без `tc-u32`/`ifb`, с 0% коллизий, IPv4/IPv6 и раздельными лимитами Download/Upload. Ставится на сервер **ноды** и шейпит порты, которые слушает Xray, чтобы один пользователь не забивал весь канал.

Возможности (подменю):
- **Активные правила / Статистика** — список правил и топ IP по трафику.
- **Добавить / изменить правило** — мастер: ID правила (0–31) → интерфейс → режим → порты (или `0` = все) → скорости DL/UL.
- **Режимы:** *Статический* (жёсткий лимит на пользователя), *Динамический* (burst → штраф → восстановление), *Общий* (одна труба делится на всех).
- **Белый список** — IP, которые игнорируют шейпер (полная скорость).
- **Перезапуск / лог / мониторинг (iftop) / полная очистка.**

Технически: eBPF-программа компилируется `clang` и грузится через `bpftool`; правила хранятся в `/etc/speedwave/traffic_shaper/rules.json` и восстанавливаются systemd-сервисом `speedwave-traffic-shaper` после перезагрузки.

> [!NOTE]
> Требуется ядро **5.4+** (XanMod из «Ускорителя ноды» подходит). Зависимости — `clang`, `llvm`, `libbpf-dev`, `bpftool`, заголовки ядра (несколько сотен МБ) — на маленьких VPS освободите место заранее. Шейпер берёт root-qdisc (`fq`) выбранного интерфейса (совместимо с BBR); при «Полной очистке» qdisc снимается. Архитектуры: x86_64 и arm64.

## Режимы развёртывания

**1. Один сервер** — панель и XRAY-нода на одной машине. Для компактных установок с умеренным трафиком.

**2. Распределённый** — отдельно сервер панели (центр управления) и сервер ноды (XRAY + заглушка SelfSteal для VLESS REALITY).

При установке ноды доступны два варианта:
- **С selfsteal-сайтом** — домен + TLS + маскировочная страница «IP заблокирован».
- **Без selfsteal-домена** — только контейнер ноды, без сайта/домена/сертификата.

### Требования к доменам

Подготовьте три домена/поддомена:
1. **Панель управления** — доступ к интерфейсу
2. **Страница подписок** — раздача конфигураций клиентам
3. **Заглушка SelfSteal** — маскировочный сайт на сервере ноды

### Примеры DNS

**Один сервер (панель + нода вместе)**

| Тип   | Имя               | Значение        | Proxy    |
|-------|-------------------|-----------------|----------|
| A     | example.com       | your_server_ip  | DNS only |
| CNAME | panel.example.com | example.com     | DNS only |
| CNAME | sub.example.com   | example.com     | DNS only |
| CNAME | node.example.com  | example.com     | DNS only |

> Запись `node.example.com` необязательна — для SelfSteal можно использовать корневой домен.

**Распределённый (панель и нода на разных серверах)**

| Тип   | Имя               | Значение        | Proxy    |
|-------|-------------------|-----------------|----------|
| A     | example.com       | panel_server_ip | DNS only |
| CNAME | panel.example.com | example.com     | DNS only |
| CNAME | sub.example.com   | example.com     | DNS only |
| A     | node.example.com  | node_server_ip  | DNS only |

## Защита доступа к панели

NGINX закрывает панель аутентификацией по URL-параметру:

```
https://panel.example.com/auth/login?<SECRET_KEY>=<SECRET_KEY>
```

Параметр выставляет cookie (`<SECRET_KEY>` = `<SECRET_KEY>`). Без правильной cookie и параметра панель отдаёт пустую страницу или 404. Это защищает от обнаружения сканированием хоста, перебора путей и брутфорса.
